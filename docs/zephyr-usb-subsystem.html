⚙️ 3. device_next 架构深度分析

3.1 线程化事件驱动模型

device_next 使用独立内核线程处理 USB 事件，区别于 legacy 的 workqueue 模式：

static K_KERNEL_STACK_DEFINE(usbd_stack, CONFIG_USBD_THREAD_STACK_SIZE);
static struct k_thread usbd_thread_data;

/* UDC 中断 → 消息队列 → 线程处理 */
K_MSGQ_DEFINE(usbd_msgq, sizeof(struct udc_event),
              CONFIG_USBD_MAX_UDC_MSG, 4);

/* 事件类型 */
enum udc_event_type {
    UDC_EVT_BUS_RESET,       // USB 总线复位
    UDC_EVT_BUS_SUSPEND,     // 总线挂起
    UDC_EVT_BUS_RESUME,      // 总线唤醒
    UDC_EVT_SETUP_RECEIVED,  // SETUP 包到达
    UDC_EVT_EP_REQUEST,      // 端点传输完成
    UDC_EVT_VBUS_READY,      // VBUS 就绪
    UDC_EVT_VBUS_REMOVED,    // VBUS 断开
    UDC_EVT_ERROR,           // 错误
};

线程在 usbd_core.c 中初始化，通过 k_msgq_get 等待事件，调用 usbd_event_handler 分发处理。

3.2 类驱动注册机制

类驱动通过 USBD_DEFINE_CLASS 宏注册，利用链接器节(section)迭代实现自动发现：

USBD_DEFINE_CLASS(class_name, &class_api, &private_data, &vendor_reqs);
// 展开为:
static struct usbd_class_data class_name = {
    .name   = "class_name",
    .api    = class_api,
    .v_reqs = vendor_reqs,
    .priv   = private_data,
};
static STRUCT_SECTION_ITERABLE_ALTERNATE(usbd_class_fs, usbd_class_node, ...) = {
    .c_data = &class_name,
};

所有类实例通过 STRUCT_SECTION_FOREACH_ALTERNATE(usbd_class_fs, ...) 自动迭代，无需手动注册数组。

类 API (struct usbd_class_api) 定义了 12 个回调：

init — 初始化
enable — 配置生效
disable — 配置禁用
update — 接口更新
request — 传输完成
get_desc — 获取描述符
control_to_host — 控制 IN
control_to_dev — 控制 OUT
feature_halt — 端点停止
suspended — 挂起
resumed — 唤醒
shutdown — 关闭

3.3 初始化流程

// 1. 定义设备上下文
USBD_DEVICE_DEFINE(my_usbd, DEVICE_DT_GET(DT_NODELABEL(zephyr_udc0)),
                   0x2fe3, 0x0001);

// 2. 定义字符串描述符
USBD_DESC_LANG_DEFINE(my_lang);
USBD_DESC_MANUFACTURER_DEFINE(my_mfr, "My Company");
USBD_DESC_PRODUCT_DEFINE(my_prod, "My Device");

// 3. 定义配置
USBD_CONFIGURATION_DEFINE(my_config, USB_SCD_SELF_POWERED, 125, &fs_cfg_desc);

// 4. 注册类驱动 (自动通过链接器节迭代)
// 类实例通过 USBD_DEFINE_CLASS 宏静态注册

// 5. 初始化
struct usbd_context *ctx = my_usbd;
usbd_init(ctx);       // ← usbd_init.c: 初始化配置、接口、端点、类
usbd_enable(ctx);     // ← 使能 UDC 控制器

3.4 描述符管理

device_next 提供了一套 类型安全的描述符构建 API，不再需要手动拼字节：

宏作用
USBD_DEVICE_DEFINE(ctx, dev, vid, pid)定义设备上下文
USBD_DESC_LANG_DEFINE(name)语言 ID 描述符
USBD_DESC_MANUFACTURER_DEFINE(name, str)厂商字符串
USBD_DESC_PRODUCT_DEFINE(name, str)产品字符串
USBD_DESC_SERIAL_NUMBER_DEFINE(name)序列号 (HWINFO 派生)
USBD_CONFIGURATION_DEFINE(name, attr, mpower, desc)配置描述符
USBD_DESC_BOS_DEFINE(name, size, caps)BOS 能力描述符

描述符在 usbd_desc.c 中管理，支持 FS/HS 双速度描述符自动切换。

🔌 4. 类驱动详解

HID 人机交互设备

源码: usbd_hid.c + usbd_hid_api.c

支持 IN/OUT 中断端点。应用通过 hid_device_register() 注册报告描述符和回调：

  • iface_ready — 接口就绪
  • get_report / set_report — 控制端点报告
  • output_report — 中断 OUT 报告
  • input_report_done — 输入报告发送完成
  • hid_device_submit_report() — 提交输入报告
CDC ACM 虚拟串口

源码: usbd_cdc_acm.c

通过 uart-bridge 驱动实现全双工 CDC ACM ↔ UART 桥接。

DT 配置：

&zephyr_udc0 {
    cdc_acm_uart0: cdc_acm_uart0 {
        compatible = "zephyr,cdc-acm-uart";
    };
};
/ {
    uart-bridge0 {
        compatible = "zephyr,uart-bridge";
        peers = <&cdc_acm_uart0 &uart1>;
    };
};
MSC 大容量存储

源码: usbd_msc.c + usbd_msc_scsi.c

实现 SCSI 命令集 + BULK-ONLY 传输 (BBB)。配合 USBD_DEFINE_MSC_LUN() 注册磁盘。

磁盘通过 disk_access_* API 访问，支持 RAM disk、SD 卡、Flash 等后端。

// 注册 RAM disk 为 LUN
USBD_DEFINE_MSC_LUN(0, "RAMDISK",
                    "VENDOR", "PRODUCT", "1.0");
DFU 固件升级

源码: usbd_dfu.c + usbd_dfu_flash.c

USB DFU 1.1 标准实现，支持 runtime/DFU 双模式。

DFU mode 时通过 Flash API 擦写内部/外部 Flash。

UAC2 USB Audio 2.0

源码: usbd_uac2.c

支持异步等时端点，多通道音频流。适用于 USB 麦克风/音箱设备。

UVC USB Video

源码: usbd_uvc.c

USB Video Class 实现，支持等时视频流传输。

📟 5. UDC 驱动层 —udc_*

UDC (USB Device Controller) 驱动是 device_next 栈的硬件抽象层。所有 UDC 驱动实现统一的 struct udc_api 接口：

核心 API
init — 初始化控制器
enable — 使能
disable — 禁用
set_address — 设地址
ep_set_halt — 端点停止
ep_enqueue — 提交传输
能力
caps.hs — 高速支持
caps.rwup — 远程唤醒
caps.can_detect_vbus — VBUS 检测
caps.mps0 — EP0 最大包
caps.addr_before_status — 地址时序

🚀 9. 快速开始

最小 HID 设备示例

// prj.conf
CONFIG_USB_DEVICE_STACK_NEXT=y
CONFIG_USBD_HID_SUPPORT=y
CONFIG_SAMPLE_USBD_VID=0x2fe3
CONFIG_SAMPLE_USBD_PID=0x0007

// app.overlay
&zephyr_udc0 {
    hid_dev0 {
        compatible = "zephyr,hid-device";
        in-report-size = 64;
        in-polling-period-us = 1000;
    };
};

// main.c
#include <zephyr/usb/usbd.h>
#include <zephyr/usb/class/usbd_hid.h>

LOG_MODULE_REGISTER(main, LOG_LEVEL_INF);

static const uint8_t report_desc[] = {
    0x06, 0x00, 0xFF,  // Vendor defined
    0x09, 0x01, 0xA1, 0x01,
    0x15, 0x00, 0x26, 0xFF, 0x00,
    0x75, 0x08, 0x95, 0x40,  // 64 byte report
    0x09, 0x01, 0x81, 0x02,
    0x09, 0x01, 0x91, 0x02,
    0xC0
};

static const struct hid_device_ops ops = {
    .set_report = my_set_report,
};

int main(void)
{
    struct usbd_context *ctx = sample_usbd_init_device(msg_cb);
    hid_device_register(hid_dev, report_desc, sizeof(report_desc), &ops);
    usbd_enable(ctx);
    return 0;
}
